# Cocktail AI App

## Mission

Cocktail AI App is a Cloud & AI Engineering portfolio project.

The objective is to demonstrate professional software engineering practices while building a production-quality, cloud-native cocktail application using AWS and AI.

Code quality, maintainability, testing, and documentation are considered features.

---

## Engineering Philosophy

- Prefer maintainable solutions over clever ones.
- Prefer small, incremental improvements.
- Keep architecture simple until complexity is justified.
- Avoid unnecessary abstraction.
- Documentation is a first-class deliverable and should remain concise, accurate and valuable.
- Code quality, testing and documentation are considered part of every feature.

---

## Engineering Role

When working on this repository:

- Think like an experienced software engineer.
- Consider the wider impact of changes across the project.
- Review affected code, tests, documentation and configuration before implementing changes.
- Think beyond the immediate file being modified.
- Explain significant architectural decisions and trade-offs.
- Verify your work before considering a task complete.

---

## Technology Stack

- Backend: Python, FastAPI
- Cloud: AWS
- Database: DynamoDB
- Authentication: Amazon Cognito (planned)
- AI: OpenAI API (planned)
- Frontend: React (planned)

---

## Coding Standards

- Use type hints.
- Use structured logging.
- Write regression tests where appropriate.
- Keep services cohesive and focused.
- Prefer readability over cleverness.

---

## Engineering Workflow

For significant changes:

1. Review the current implementation and existing documentation.
2. Identify affected code, tests, documentation and configuration.
3. Produce a brief implementation plan.
4. If architectural or structural changes are proposed, obtain approval before implementation.
5. Implement the agreed changes.
6. Update documentation where changes materially improve understanding.
7. Verify the implementation.
8. Summarise:
   - What changed
   - Why it changed
   - How it was verified

When updating documentation:

- Clearly distinguish **Current Implementation** from **Future Direction**.
- Do not describe planned functionality as implemented.
- Preserve relevant future architectural direction where appropriate.

---

## Developer Experience

Small improvements that improve developer or user experience are encouraged when they:

- are low risk
- are easy to verify
- improve overall project quality
- do not introduce unnecessary complexity

---

## Definition of Done

Before completing a task:

- Existing tests pass.
- New functionality is tested where appropriate.
- Documentation has been reviewed and updated if required.
- Logging and error handling have been considered.
- The implementation remains architecturally consistent.
- No unnecessary duplication has been introduced.
- Summarise the implementation and important design decisions.

## Repository Governance

`AGENTS.md` defines the long-term engineering principles for this repository.

Do not modify `AGENTS.md` as part of normal implementation work.

If you identify an opportunity to improve these engineering principles:

- Do not edit this file.
- Instead, recommend the proposed change in your final summary under **Suggested AGENTS.md Updates**.
- Explain the reasoning for the recommendation.
- Wait for explicit user approval before making any changes to `AGENTS.md`.

Only recommend changes that improve future engineering across the repository. Do not add feature-specific or temporary guidance.

## Documentation Maintenance

When implementing a completed feature:

- Update CHANGELOG.md if the feature changes application behaviour, architecture or developer workflows.
- Update the roadmap if milestones change.
- Update architecture documentation if project structure changes.
- Update setup.md if installation or configuration changes.
